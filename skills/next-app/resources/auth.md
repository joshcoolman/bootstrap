# Auth — self-hosted, no vendor, no database

Everything here is code the app owns. No auth service, no signup route, no
OAuth, no password reset. ~200 lines, all standard library: `node:crypto`
scrypt for passwords, Web Crypto HMAC for the session cookie.

**This is allowlist auth.** A short, known set of people can sign in. Access is
granted out-of-band by an operator, never self-served by a visitor. Anyone in
the allowlist gets identical access — the app does not distinguish between them
beyond an opaque id.

If an app ever grows a real user base — signup, reset, verification, OAuth,
roles — that is the signal to adopt `better-auth`, not to extend this. Say so
plainly rather than bolting features on.

Lifted from a production app (`bootsy`), with two deliberate changes flagged
below.

## Read this first — the two things that break

**1. The Edge/Node split.** `proxy.ts` runs on the Edge runtime. It must import
`./session` directly and must never import `index.ts`, because that barrel
re-exports `credentials.ts`, which pulls in `node:crypto` and breaks the Edge
bundle. This layout keeps the two halves in separate modules so the split is
structural rather than a comment someone has to remember.

**2. Dots in the session value.** The cookie is `userId.issuedAtMs.signature`,
split on `.`. Any userId containing a dot breaks the parse — and an email is the
obvious id to reach for. The `env` backend therefore derives an **opaque id by
hashing the email**. bootsy uses raw UUIDs and is safe by accident; do not carry
that assumption into a backend where ids are emails.

## Credential format

`AUTH_USERS` is a comma-separated list of `email:salt:hash` entries. Salt and
hash are hex, so the only `:` characters are the two separators, and emails
cannot contain `:`.

```
AUTH_USERS=josh@example.com:9f86d0…:c2b7d1…,friend@example.com:4a7d1e…:ef2d12…
```

Generated by `pnpm auth:add-user`, which prints a line to paste. Adding a person
is adding an entry and redeploying; removing one is deleting the entry.

`AUTH_SESSION_SECRET` is any long random string — `openssl rand -hex 32`.
Changing it invalidates every existing session, which is the closest thing this
design has to a global sign-out.

## `src/features/auth/hash.mjs`

Plain JavaScript, not TypeScript, and this is deliberate: it is imported by both
the app (`credentials.ts`) and the user-add script (`scripts/add-user.mjs`).
bootsy kept two copies with a comment warning they must stay identical — drift
there is a silent auth failure, where a generated credential simply never
verifies. One module removes the class of bug entirely.

```js
// Shared by src/features/auth/credentials.ts and scripts/add-user.mjs.
// Plain .mjs so the Node script can import it without a build step — one
// implementation instead of two that must be kept identical by hand.
import { createHash, randomBytes, scrypt, timingSafeEqual } from 'node:crypto'
import { promisify } from 'node:util'

const scryptAsync = promisify(scrypt)
const KEY_LENGTH = 64

// Fixed valid-shaped hash to verify against when no entry matches the given
// email, so a login attempt takes the same time whether or not the email is
// registered (no timing side-channel revealing who has access).
export const DUMMY_HASH = `${'0'.repeat(32)}:${'0'.repeat(128)}`

export function normalizeEmail(email) {
  return email.trim().toLowerCase()
}

// Opaque, stable, dot-free user id. Dot-free matters: the session cookie is
// `userId.issuedAtMs.signature` split on '.', so an email as the id would
// break the parse. Hashing also keeps the email out of the cookie.
export function userIdForEmail(email) {
  return createHash('sha256').update(normalizeEmail(email)).digest('hex').slice(0, 16)
}

export async function hashPassword(password) {
  const salt = randomBytes(16).toString('hex')
  const derivedKey = await scryptAsync(password, salt, KEY_LENGTH)
  return `${salt}:${derivedKey.toString('hex')}`
}

export async function matchesHash(password, storedHash) {
  const [salt, hashHex] = storedHash.split(':')
  if (!salt || !hashHex) return false

  const derivedKey = await scryptAsync(password, salt, KEY_LENGTH)
  const storedKey = Buffer.from(hashHex, 'hex')
  if (derivedKey.length !== storedKey.length) return false

  return timingSafeEqual(derivedKey, storedKey)
}
```

## `src/features/auth/session.ts`

The Edge-safe half. Web Crypto only — no `node:*` imports, because middleware
imports this module directly.

```ts
// Edge-safe: Web Crypto only, no `node:*` imports. Middleware imports this
// module directly (never `./index`, which re-exports `./credentials` and
// pulls in node:crypto).

export const SESSION_COOKIE_NAME = '<app>_session'
export const SESSION_MAX_AGE_SECONDS = 60 * 60 * 24 * 30 // 30 days

function getSessionSecret(): string {
  const secret = process.env.AUTH_SESSION_SECRET
  if (!secret) throw new Error('AUTH_SESSION_SECRET is not set')
  return secret
}

async function hmacHex(message: string, secret: string): Promise<string> {
  const key = await crypto.subtle.importKey(
    'raw',
    new TextEncoder().encode(secret),
    { name: 'HMAC', hash: 'SHA-256' },
    false,
    ['sign'],
  )
  const signature = await crypto.subtle.sign('HMAC', key, new TextEncoder().encode(message))
  return Array.from(new Uint8Array(signature))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('')
}

// Constant-time compare of two equal-length hex strings (Web Crypto has no
// timing-safe raw compare outside `subtle.verify`).
function timingSafeEqualHex(a: string, b: string): boolean {
  if (a.length !== b.length) return false
  let diff = 0
  for (let i = 0; i < a.length; i++) {
    diff |= a.charCodeAt(i) ^ b.charCodeAt(i)
  }
  return diff === 0
}

export type Session = { userId: string; issuedAtMs: number }

export async function createSessionValue(userId: string): Promise<string> {
  const issuedAtMs = Date.now()
  const signature = await hmacHex(`${userId}.${issuedAtMs}`, getSessionSecret())
  return `${userId}.${issuedAtMs}.${signature}`
}

export async function verifySessionValue(value: string | undefined): Promise<Session | null> {
  if (!value) return null
  const [userId, issuedAtRaw, signature] = value.split('.')
  if (!userId || !issuedAtRaw || !signature) return null

  const issuedAtMs = Number(issuedAtRaw)
  if (!Number.isFinite(issuedAtMs)) return null

  let expectedSignature: string
  try {
    expectedSignature = await hmacHex(`${userId}.${issuedAtRaw}`, getSessionSecret())
  } catch {
    return null
  }
  if (!timingSafeEqualHex(signature, expectedSignature)) return null

  if (Date.now() - issuedAtMs >= SESSION_MAX_AGE_SECONDS * 1000) return null
  return { userId, issuedAtMs }
}

export function sessionCookieOptions() {
  return {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax' as const,
    path: '/',
    maxAge: SESSION_MAX_AGE_SECONDS,
  }
}
```

**The session is stateless** — no sessions table, no lookup. State that
consequence in the app's docs: there is no server-side revocation. Sign-out
clears the cookie on that browser; a stolen cookie stays valid until it expires
or `AUTH_SESSION_SECRET` changes.

## `src/features/auth/credentials.ts` — env backend (default)

```ts
import 'server-only'
import { DUMMY_HASH, matchesHash, normalizeEmail, userIdForEmail } from './hash.mjs'

// AUTH_USERS is a comma-separated list of `email:salt:hash` entries. Salt and
// hash are hex, so the only ':' characters are the two separators.
function getAllowlist(): Map<string, string> {
  const raw = process.env.AUTH_USERS ?? ''
  const entries = new Map<string, string>()

  for (const entry of raw.split(',')) {
    const trimmed = entry.trim()
    if (!trimmed) continue

    const [email, salt, hash] = trimmed.split(':')
    if (!email || !salt || !hash) continue

    entries.set(normalizeEmail(email), `${salt}:${hash}`)
  }
  return entries
}

export async function verifyCredentials(email: string, password: string): Promise<string | null> {
  const storedHash = getAllowlist().get(normalizeEmail(email))

  // Always run the comparison, even with no match, so timing doesn't reveal
  // which emails are on the allowlist.
  const isMatch = await matchesHash(password, storedHash ?? DUMMY_HASH)
  return isMatch && storedHash ? userIdForEmail(email) : null
}
```

The allowlist is parsed per call rather than cached at module load, so a changed
env var takes effect on the next request in dev without a restart. It's a
handful of string splits over two or three entries — not worth memoizing.

## `credentials.ts` — db backend (only when Postgres already exists)

Same contract, different lookup. Requires a `users` table (`id uuid primary key
default gen_random_uuid()`, `email text unique not null`, `password_hash text
not null`) and `pnpm auth:add-user` inserting rather than printing.

```ts
import 'server-only'
import { DUMMY_HASH, matchesHash, normalizeEmail } from './hash.mjs'
import { sql } from './db'

export async function verifyCredentials(email: string, password: string): Promise<string | null> {
  const [user] = await sql<{ id: string; passwordHash: string }[]>`
    select id, password_hash from users where email = ${normalizeEmail(email)}
  `
  const isMatch = await matchesHash(password, user?.passwordHash ?? DUMMY_HASH)
  return isMatch && user ? user.id : null
}
```

Here the id is a UUID, so no hashing is needed — UUIDs contain no dots.

## `src/features/auth/current-user.ts`

```ts
import { cookies } from 'next/headers'
import 'server-only'
import { SESSION_COOKIE_NAME, verifySessionValue } from './session'

// Server actions / RSCs only — never imported by proxy.ts, which reads
// ./session directly and only needs validity, not identity.
export async function getCurrentUserId(): Promise<string> {
  const cookieStore = await cookies()
  const session = await verifySessionValue(cookieStore.get(SESSION_COOKIE_NAME)?.value)
  if (!session) throw new Error('Not authenticated')
  return session.userId
}
```

**It throws rather than returning null**, deliberately. Data access should never
silently proceed unauthenticated because a caller forgot to check. The
middleware is the gate; this is the backstop, and defence in depth means not
trusting the gate alone.

## `src/features/auth/actions.ts`

```ts
'use server'

import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import { verifyCredentials } from './credentials'
import { createSessionValue, SESSION_COOKIE_NAME, sessionCookieOptions } from './session'

export type LoginState = { error: string | null; email: string }

export async function login(_prev: LoginState, formData: FormData): Promise<LoginState> {
  const email = String(formData.get('email') ?? '')
  const password = String(formData.get('password') ?? '')

  const userId = await verifyCredentials(email, password)
  // One message for both cases — never reveal whether the email is known.
  if (!userId) return { error: 'Invalid email or password', email }

  const cookieStore = await cookies()
  cookieStore.set(SESSION_COOKIE_NAME, await createSessionValue(userId), sessionCookieOptions())
  redirect('/dashboard')
}

export async function logout(): Promise<void> {
  const cookieStore = await cookies()
  cookieStore.set(SESSION_COOKIE_NAME, '', { ...sessionCookieOptions(), maxAge: 0 })
  redirect('/login')
}
```

`LoginState` carries `email` back deliberately. React 19's `useActionState`
resets *uncontrolled* form fields after every action call, success or failure —
so a rejected login would otherwise wipe what the user typed. Echo it back and
set it as the input's `defaultValue`, or make the field controlled.

## Tests

Two suites, both required by the build gates.

### `session.test.ts`

Cover: round-trip returns the userId; a tampered signature is rejected; a
tampered userId with the *original* signature is rejected; an expired timestamp
with a *valid* signature is rejected; malformed input returns null rather than
throwing.

The expired case must sign a genuinely old timestamp with the real secret —
otherwise it passes for the wrong reason (bad signature rather than expiry).

```ts
import { beforeAll, describe, expect, it } from 'vitest'
import { createSessionValue, verifySessionValue } from '<alias>/features/auth/session'

const USER_ID = 'a1b2c3d4e5f60718'

beforeAll(() => {
  process.env.AUTH_SESSION_SECRET = 'test-secret'
})

async function hmacHex(message: string, secret: string): Promise<string> {
  const key = await crypto.subtle.importKey(
    'raw', new TextEncoder().encode(secret),
    { name: 'HMAC', hash: 'SHA-256' }, false, ['sign'],
  )
  const sig = await crypto.subtle.sign('HMAC', key, new TextEncoder().encode(message))
  return Array.from(new Uint8Array(sig)).map((b) => b.toString(16).padStart(2, '0')).join('')
}

describe('session', () => {
  it('verifies a freshly created value and returns its userId', async () => {
    const value = await createSessionValue(USER_ID)
    expect(await verifySessionValue(value)).toEqual({
      userId: USER_ID, issuedAtMs: expect.any(Number),
    })
  })

  it('rejects a tampered signature', async () => {
    const [userId, issuedAt, sig] = (await createSessionValue(USER_ID)).split('.')
    const flipped = `${sig!.slice(0, -1)}${sig!.at(-1) === '0' ? '1' : '0'}`
    expect(await verifySessionValue(`${userId}.${issuedAt}.${flipped}`)).toBeNull()
  })

  it('rejects a tampered userId even with the original signature', async () => {
    const [, issuedAt, sig] = (await createSessionValue(USER_ID)).split('.')
    expect(await verifySessionValue(`ffffffffffffffff.${issuedAt}.${sig}`)).toBeNull()
  })

  it('rejects an expired timestamp even with a valid signature', async () => {
    const expired = Date.now() - 31 * 24 * 60 * 60 * 1000
    const sig = await hmacHex(`${USER_ID}.${expired}`, 'test-secret')
    expect(await verifySessionValue(`${USER_ID}.${expired}.${sig}`)).toBeNull()
  })

  it('rejects malformed input without throwing', async () => {
    expect(await verifySessionValue(undefined)).toBeNull()
    expect(await verifySessionValue('')).toBeNull()
    expect(await verifySessionValue('not-a-valid-value')).toBeNull()
  })
})
```

### `credentials.test.ts`

Build an `AUTH_USERS` value with `hashPassword` at setup, then assert: the
correct password returns a stable id; the wrong password returns null; an
unknown email returns null; a second entry in the list also resolves; a
malformed entry is skipped rather than throwing; email matching is
case-insensitive.

**The round-trip is the important one** — hash via `hash.mjs`, verify through
`verifyCredentials`. That is the mechanism that catches drift between the script
and the app, and it is why `hash.mjs` is shared rather than duplicated.

## `scripts/add-user.mjs`

Prints a credential line. Writes no files — pasting is deliberate, so the
operator sees exactly what is being granted and where it lands.

```js
// Prints an AUTH_USERS entry for a new person. Provisioning IS the allowlist —
// there is no signup route, so this is the only way to grant access.
// Usage: pnpm auth:add-user '<email>' '<password>'
import { hashPassword, normalizeEmail, userIdForEmail } from '../src/features/auth/hash.mjs'

const [email, password] = process.argv.slice(2)

if (!email || !email.includes('@') || !password || password.length < 8) {
  console.error("Usage: pnpm auth:add-user '<email>' '<password>'  (password min 8 chars)")
  process.exit(1)
}

const entry = `${normalizeEmail(email)}:${await hashPassword(password)}`

console.log('\n  Add this to AUTH_USERS (comma-separated if there are others):\n')
console.log(`  ${entry}\n`)
console.log(`  user id: ${userIdForEmail(email)}`)
console.log('  Set it in .env.local and on the host, then redeploy.\n')
```

Add to `package.json`:

```json
"auth:add-user": "node scripts/add-user.mjs"
```

Note the name — never `setup`, which collides with a pnpm built-in.

**The password is passed as an argument**, so it lands in shell history. Say so
in the handoff and suggest a leading space (with `HIST_IGNORE_SPACE`) or
clearing the entry afterwards. Do not switch to an interactive prompt that
echoes — that is exactly how a credential ends up in a screenshot.

## Adaptation slots

| Slot | Filled from |
| --- | --- |
| `<alias>` | discovery — the repo's import alias (`#/`, `@/`, …) |
| `<app>` | the app name, for `SESSION_COOKIE_NAME` |
| `<features-dir>` | discovery — where features live |

Cookie name must be app-specific. Two apps on `localhost` sharing a cookie name
will overwrite each other's sessions during development.
